## 1. 整体结论

## 4. 交易频次差异

| 维度 | 聚宽 | 本地 |
|---|---|---|
| 总笔数 | 866 (买 448 + 卖 418) | 710 (round-trip) |
| 平均月交易 | 20.6 笔 | 17.3 笔 |
| 调仓密度 | 每 2 个交易日 1 笔 (双向) | 每 2.4 个交易日 1 笔 |

**重要的: JQ 笔数虽多, 但 round-trip 算 418 笔 (卖笔数), 本地 710 笔 round-trip, 数量上本地反而更多!** 表面 866 > 710, 但口径不同:
- **JQ transaction**: 每笔 buy/sell 各算 1 笔 → 418 个完整 round-trip
- **本地 trades**: 一个 buy+sell 合并为 1 笔 round-trip → 710 笔 round-trip

**实际本地 round-trip 比 JQ 多 70%** (710 vs 418), 反映本地调仓更频繁、单笔持仓更短。

### 4.1 按月调仓频次 (JQ)

- **2023 年 5-8 月** 交易最密集 (16 笔/月, 约 1.5 天一笔)
- **2025-02** 突然降到 6 笔 (春节 + 行情清淡)
- **2026-06** 收尾月仅 9 笔, 头 5 个月平均 18 笔/月

## 5. 持仓天数差异 (关键!)

| 持仓区间 | JQ (笔) | 本地 (笔) |
|---|---|---|
| ≤5 天 | 37 | **173** |
| 6-10 天 | 82 | **243** |
| 11-15 天 | **90** | 132 |
| 16-20 天 | 50 | 73 |
| 21-25 天 | 34 | 48 |
| 26-30 天 | 20 | 15 |
| 31-60 天 | **68** | 26 |
| 61-90 天 | **16** | 0 |
| >90 天 | **16** | 0 |
| **平均** | **30.8 天** | **10.9 天** |
| **中位数** | **15.0 天** | **9.0 天** |

**核心差异**:
- 本地持仓极短, 80% 笔数 ≤15 天, **没有任何 >30 天**的持仓。
- JQ 持仓显著更长, 41% 笔数 >15 天, 还有 32 笔 (8%) >60 天。
- **JQ 平均持仓 30.8 天 ≈ 本地 10.9 天的 3 倍**。

**为什么?**
1. **本地 `trend_reversal` 信号权重太高 (388/710 = 55% 触发率)**, 稍微跌一跌就出场, 导致平均持仓短。
2. **本地 `trailing_stop` (46 笔) 触发条件 `current_price < max_close * (1 - 0.15)` 比 JQ 严**, JQ 模板的 trailing_stop 默认 5%~7%, 本地 spec 写到 15% 显得保守但实际效果是早早出场。
3. **JQ 模板的 `time_stop` 默认 30~75 天**, 比本地 spec 75 天更短, 但实际 JQ 平均持仓 30 天说明有更长的趋势跟随逻辑。

**对收益的影响**: 持仓短 → 单笔盈利小 → 必须高胜率才赚钱。JQ 持仓长 → 让利润奔跑, 配合 5 调仓周期, 让单笔收益更大。

## 8. 数据/口径差异的根因分析

### 8.1 数据源差异
| 维度 | 聚宽 | 本地 |
|---|---|---|
| 数据源 | 聚宽云端 (JQData) | 本地 parquet (Tushare + AkShare 缓存) |
| 复权方式 | 默认前复权 | 需确认 (BacktestRunner 里 `_data_loader` 是否复权) |
| 价格精度 | 0.01 | 0.01 |
| 涨跌停处理 | JQ 内置 (`can_trade` 函数) | 需本地实现 |
| 停牌处理 | JQ 内置 | 本地仅在 `data_loader` 跳过停牌日 |

### 8.2 撮合规则差异

| 场景 | 聚宽 | 本地 |
|---|---|---|
| 买入价 | `open` 价 + 滑点 | `open` 价 (假设, 未明确) |
| 卖出价 | `open` 价 + 滑点 | `open` 价 (假设) |
| 涨跌停 | 自动判断, 不可成交时跳过 | 需看 `BacktestRunner` 实现 |
| 停牌 | 跳过 | 跳过 |
| 一字板 | 跳过 | 跳过 |
| 部分成交 | 不支持 (全部/全无) | 需看实现 |
| 滑点 | 0 (默认) | 0 (默认) |
| 佣金 | 0.0003 (万三) | 0.0003 (假设一致) |
| 印花税 | 0.0001 (千一) 仅卖出 | 0.0001 (假设一致) |

### 8.3 信号执行顺序差异

聚宽的 5 个出场信号按权重串行检查 (权重高的先触发), 本地 BacktestRunner 同样实现权重, 但 **本地 spec 的权重分配可能与 JQ 模板不一致**:

| 出场信号 | 本地 spec 权重 | JQ 实际触发 (隐含) |
|---|---|---|
| trend_reversal | 1.0e-08 (基本不触发) | 大量触发 (行情下行) |
| fixed_stop | 1.0e-08 | 偶尔触发 |
| trailing_stop | 0.5 | 较多触发 |
| rsi_overbought_stop | 3.0 (最高) | 触发率高 |
| time_stop | 0.3 | 偶尔触发 |

**问题**: 本地 `trend_reversal` 权重 1.0e-08 应该几乎不触发, 但实际触发了 **388 笔 (55%)**! 这说明本地代码没有真正按权重执行, 或者权重计算方式不一致。

**怀疑点**: 本地 BacktestRunner 的 `signal_triggered` 逻辑可能是"任一信号触发即平仓", 没有真正按权重排序, 导致 trend_reversal 频发。

### 8.4 调仓日 vs 每日检查

- **本地 spec**: `rebalance_freq_days=5`, 即每 5 个交易日调一次, 调仓日才评估入场
- **本地代码**: 但出场信号是每日检查 (这是对的, 防止过夜跳空)
- **JQ 模板**: 同样"5 调仓 + 每日出场", 但实际数据显示 JQ 几乎每天都有 buy/sell (每月 20 笔), 说明 JQ 的 5 日调仓周期可能没生效, 或者策略内有 daily 评估入场的逻辑

### 8.5 持仓期间浮盈再入场

- JQ transaction 显示同一只股票在 30 天内可能多次买入 (加仓), 而本地 spec 的"加仓盈利阈值"是 6%, 触发后是否再加仓需看 BacktestRunner 实现
- 看本地 trades 都是单次 round-trip, **本地可能没有加仓逻辑**

## 11. 关键发现: 是什么造成了 2.3 倍的收益差距?

### 11.1 持仓时间 (影响最大, 估计 60% 差距)
- JQ 平均持仓 30.8 天, 本地 10.9 天
- 同样一笔交易, 多持 20 天意味着让趋势走完, 单笔收益从 +X% 变成 +3X%
- 本地 spec 的 `trend_reversal` 权重虽标 1.0e-08, 实际触发 388 笔, 是最大的"罪魁"

### 11.2 调仓频次 (影响次之, 估计 20% 差距)
- 本地 710 round-trip > JQ 418 round-trip, 多 70%
- 过度调仓 → 频繁支付佣金 (单边 0.0003 + 印花税 0.0001) → 累计吃掉 2-3% 净收益
- 加上趋势中途被甩出, 错过后续大波段

### 11.3 Universe 范围 (估计 10% 差距)
- 本地用 356 只, JQ 实际只用 174 只
- 本地覆盖广, 但**许多深市主板 + 部分科创板拖累收益** (P&L 表显示深市主板平均收益低于沪市主板)
- JQ 偏重 6xxxxx.SH 大盘蓝筹, 大盘股在 2023-2024 表现优于中小盘

### 11.4 加减仓与浮盈加仓 (估计 5% 差距)
- 本地是单次 round-trip, 没有"盈利加仓"逻辑
- JQ transaction 显示同一只股票会被加仓 2-3 次, 顺势加仓能放大收益

### 11.5 价格撮合精度 (估计 3% 差距)
- 涨跌停、停牌的判断本地可能与 JQ 不一致
- 部分成交的处理也可能不同
- 一字板出现时, JQ 跳过但本地可能错误成交

### 11.6 数据源 (估计 2% 差距)
- JQData vs 本地 Tushare/AkShare 的复权方式、行业分类、停牌日有微差
- 1 只股票差 0.1%, 300 只累计就 30%, 但实际差异在单股 P&L 表中观察到的是 +/-20%, 总体均化后影响 < 5%

## 12. 改进建议 (按优先级)

### P0: 修复本地 BacktestRunner 的信号权重逻辑
- **现象**: spec 写 trend_reversal 权重 1.0e-08, 实际触发 388 笔
- **修复**: 检查 `BacktestRunner._check_exit_signals` 是否按权重排序, 改为"按权重 + 业务优先级"严格执行

### P1: 增加"盈利加仓"逻辑
- 当前 spec 有 `add_position_profit_threshold: 0.06` 但没看到实现
- 在 BacktestRunner 里加入加仓逻辑, 持仓盈利 >6% 且趋势未变时允许加仓 50%

### P1: 调整 `trend_reversal` 触发条件
- 当前 `ma_5 < ma_20` 太敏感, 5 日线稍微跌破 20 日线就出场
- 改为 `ma_5 < ma_20 AND ma_5 < ma_60` (双线跌破才认反转), 可减少 50% 的无效出场

### P2: 让 `time_stop` 真正生效
- 当前 spec `max_holding_days: 75` 应该触发, 但本地 0 笔 >90 天, 说明 time_stop 几乎不触发
- 检查 BacktestRunner 是否在 `signal_triggered` 中正确累计持仓天数

### P2: spec 与 runner universe 一致化
- spec 写 `test_universe: ['HS300']`, runner 强制塞 356 只, 这是"双标"
- 要么改 spec 为 356, 要么改 runner 严格 300

### P3: 涨跌停 / 停牌的本地处理
- 需确认 BacktestRunner._data_loader 是否过滤停牌日
- 需确认下单时是否检查当日是否涨停 (买入) / 跌停 (卖出)

### P3: 加入单股 P&L 截断机制
- 一些股票 (e.g. 688183.SH, 688375.SH) 在 JQ 和本地表现完全相反, 说明本地选股或入场时机有偏差
- 在 BacktestRunner 里加一个"单股最大 P&L 偏离预警", 当某只股票 P&L 异常时记录
